# 1.6 النشر

> الذكاء الاصطناعي يجعل كتابة الكود أسرع بعشر مرات. وإذا بقي نشر هذا الكود طقساً بطيئاً ويدوياً ومعرّضاً للخطأ يعرفه شخص واحد فقط، فإن عنق الزجاجة ينتقل ببساطة إلى الأسفل - والنشر السيئ بسرعة 10x يعني عطلاً أكبر، لا أصغر.

---

## الأهداف

- توثيق عملية النشر الحالية حتى تتوقف عن العيش في رأس شخص واحد
- تحديد نقاط الألم التي تصبح خطيرة بمجرد أن يسرّع الذكاء الاصطناعي الإنتاج
- وضع runbook مكتوب وقابل للتكرار مع سكربت `deploy.sh` بسيط
- إضافة maintenance mode ومسار rollback مُختبَر
- تمهيد الأرضية للنشر المؤتمت المربوط بـ CI الذي يأتي في Phase 2 و Phase 3

---

## لماذا النشر جزء من Phase 1

Phase 1 تدور حول تثبيت الأساس قبل تقديم الذكاء الاصطناعي. المعايير، انضباط Git، والكود النظيف لا تعني شيئاً إذا كان الطريق إلى الإنتاج فوضى.

في الوقت الحالي، يُرجّح أن الفريق ينشر بالطريقة التي تنشر بها معظم مشاريع Laravel القديمة: شخص ما يدخل عبر SSH إلى الـ server، يسحب آخر كود، يشغّل بضعة أوامر من ذاكرته، ويأمل ألا ينكسر شيء. هذا ينجح بسرعة الإنسان لأن التغييرات نادرة. ويتوقف عن النجاح في اللحظة التي يبدأ فيها الذكاء الاصطناعي بإنتاج تغييرات أسرع مما يستطيع نشر يدوي هش استيعابه.

هذه الصفحة لا تؤتمت النشر - هذا عمل Phase 2 ([CI Pipeline](../phase2/02-ci-pipeline.md)) و Phase 3 ([CI/CD Maturity](../phase3/02-cicd-maturity.md)). إنها تفعل شيئاً أبسط وأكثر إلحاحاً: **جعل العملية الحالية مكتوبة وقابلة للتكرار وقابلة للعكس.**

---

## مراجعة العملية الحالية

الخطوة الأولى هي توثيق صادق. اكتب بالضبط كيف يحدث النشر اليوم - كل أمر، بالترتيب، بما في ذلك الأوامر التي ينساها الناس.

النشر اليدوي النموذجي لـ Laravel يبدو هكذا:

| الخطوة | الأمر | كثيراً ما يُنسى؟ |
|--------|-------|------------------|
| 1. الاتصال بالـ server | `ssh deploy@server` | - |
| 2. سحب آخر كود | `git pull origin main` | - |
| 3. تثبيت الـ dependencies | `composer install --no-dev --optimize-autoloader` | أحياناً (`--no-dev`) |
| 4. تشغيل الـ migrations | `php artisan migrate --force` | **كثيراً** |
| 5. مسح / إعادة بناء الـ caches | `php artisan config:cache route:cache view:cache` | **كثيراً** |
| 6. بناء أصول الواجهة | `npm ci && npm run build` | أحياناً |
| 7. إعادة تشغيل queue workers | `php artisan queue:restart` | **دائماً تقريباً** |

> إذا لم يستطع فريقك إنتاج هذه القائمة من الذاكرة بنفس الترتيب في كل مرة، فهذه هي المشكلة التي وُجدت هذه الصفحة لحلها.

المُخرَج من هذا القسم هو مستند واحد - مُضاف للـ repo - يلتقط العملية الحقيقية، لا المثالية.

## نقاط الألم

بمجرد كتابة العملية، تصبح نقاط الضعف واضحة. هذه هي التي تتحول من "مزعجة" إلى "خطيرة" في اللحظة التي يزيد فيها الذكاء الاصطناعي حجم التغييرات وتواترها.

| نقطة الألم | الأثر اليوم | الخطر بعد أن يسرّع الذكاء الاصطناعي الإنتاج |
|------------|-------------|---------------------------------------------|
| خطوات يدوية محفوظة | خطوة منسية أحياناً | نشر أكثر ← فرص أكثر لنسيان خطوة |
| لا خطة rollback | hotfixes مُجهِدة | تغيير سيئ من الذكاء الاصطناعي بلا تراجع سريع |
| downtime أثناء النشر | المستخدمون يرون أخطاء أثناء النشر | نشر أكثر تواتراً = downtime أكثر تواتراً |
| انحراف config / env | "يعمل على staging لا على prod" | كود AI يمر بـ CI ثم يفشل على server منحرف |
| `migrate` منسي | أخطاء runtime بعد النشر | تغييرات schema من AI غير مُطبَّقة بصمت |
| النشر من جهاز مطور | غير متتبَّع، غير قابل للتكرار | لا سجل لما شُحن فعلاً |
| لا إعادة تشغيل queue | الـ workers تشغّل كوداً قديماً | jobs الخلفية تشغّل logic قديم على بيانات جديدة |

الخيط المشترك: **كل واحدة من هذه مشكلة ذاكرة بشرية.** الحل هو نقل العملية من الذاكرة إلى سكربت وقائمة تحقق.

## التحسينات

هذه خفيفة الأثر عن قصد. الهدف هو الاستقرار والقابلية للتكرار الآن - لا pipeline كامل لـ CI/CD (هذا يأتي لاحقاً).

### 1. Runbook مكتوب

أضف `DEPLOYMENT.md` للـ repo بالخطوات الدقيقة والمرتّبة من مراجعة العملية الحالية. مصدر واحد للحقيقة. أي شخص في الفريق يستطيع النشر باتباعه.

### 2. سكربت `deploy.sh` بسيط

حوّل الـ runbook إلى سكربت حتى لا تُتخطّى الخطوات أو يُعاد ترتيبها:

```bash
#!/usr/bin/env bash
set -euo pipefail   # التوقف عند أي خطأ

php artisan down --retry=15          # وضع الصيانة

git pull origin main
composer install --no-dev --optimize-autoloader
npm ci && npm run build

php artisan migrate --force          # --force = بدون سؤال تفاعلي

php artisan config:cache
php artisan route:cache
php artisan view:cache
php artisan queue:restart

php artisan up                        # العودة للعمل
```

السطر `set -euo pipefail` هو المهم: إذا فشلت أي خطوة، يتوقف السكربت بدلاً من المضي عرجاً إلى حالة نشر نصفية.

### 3. وضع الصيانة (Maintenance Mode)

`php artisan down` يضع التطبيق خلف صفحة صيانة أثناء النشر؛ و`php artisan up` يعيده. المستخدمون يرون صفحة "سنعود قريباً" نظيفة بدلاً من أخطاء 500 في منتصف النشر.

> **قاعدة عملية:** لفّ كل نشر دائماً بـ `down` / `up`. ثوانٍ قليلة من صفحة الصيانة أفضل بكثير من مستخدم يصطدم بقاعدة بيانات نصف مهاجَرة.

### 4. مسار rollback مُختبَر

نشر لا تستطيع التراجع عنه هو مقامرة. ضع tag لكل release حتى تتمكن من العودة إلى حالة معروفة-جيدة:

```bash
git tag -a v1.4.2 -m "Release 1.4.2" && git push --tags   # قبل النشر
git checkout v1.4.1 && ./deploy.sh                          # rollback
```

الـ rollback حقيقي فقط إذا شغّلته مرة فعلاً. اختبره على staging قبل أن تحتاجه في الإنتاج.

### 5. لا تنشر من جهازك الشخصي

النشر يجب أن يعمل على الـ server (أو جهاز نشر ثابت)، من `main`، باستخدام السكربت المُضاف - لا بشكل عشوائي من جهاز مطور. هذا يضمن أن ما يُنشر يطابق ما في الـ repo.

### ما يأتي لاحقاً (ليس الآن)

| التحسين | المرحلة | لماذا الانتظار |
|---------|---------|-----------------|
| releases ذرية / zero-downtime (Deployer, Envoyer) | Phase 3 | يحتاج عملية أساس مستقرة أولاً |
| النشر فقط بعد نجاح CI | Phase 2 | يتطلب [CI Pipeline](../phase2/02-ci-pipeline.md) |
| نشر مؤتمت عند الدمج إلى `main` | Phase 3 | انظر [CI/CD Maturity](../phase3/02-cicd-maturity.md) |
| ضمان environment parity | Phase 2 | انظر [2.3 Environment](../phase2/03-environment.md) |

السكربت والـ runbook اللذان تكتبهما الآن ليسا للرمي - يصبحان الخطوات الدقيقة التي يشغّلها الـ pipeline المؤتمت لاحقاً. أنت لا تبني مرتين؛ بل تبني الأساس.

## قائمة التحقق - اكتمل عند

- [ ] عملية النشر الحقيقية موثّقة خطوة بخطوة في `DEPLOYMENT.md`
- [ ] سكربت `deploy.sh` موجود ومُضاف ويستخدم `set -euo pipefail`
- [ ] كل نشر يشغّل `php artisan down` قبله و`php artisan up` بعده
- [ ] الـ migrations تعمل بـ `--force` والـ caches يُعاد بناؤها كجزء من السكربت
- [ ] queue workers يُعاد تشغيلها في كل نشر (`queue:restart`)
- [ ] الـ releases موسومة بـ tags في Git، و rollback تم اختباره على staging
- [ ] النشر يعمل على الـ server من `main`، لا من جهاز مطور
- [ ] الفريق كله يستطيع النشر باتباع الـ runbook - لا شخص واحد فقط
