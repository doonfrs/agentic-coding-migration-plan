# 1.5 Claude Code بالتفصيل

> Claude Code ليس autocomplete — إنه agent برمجة كامل يقرأ الـ codebase، يخطط التغييرات، يحرر الملفات، ينفذ الأوامر، ويكرر العملية. هذا القسم يغطي كل ميزة يحتاج فريقك لإتقانها.

> **الفصل عن 1.3:** *قرار* استخدام Claude Code والمقارنة مع البدائل موجود في [1.3 اختيار الأدوات](03-tooling-selection.md). *المرجع التفصيلي للميزات* موجود هنا.

---

## الأهداف

- فهم كل ميزات Claude Code الرئيسية قبل استخدامه على كود الـ production
- معرفة متى تستخدم الـ CLI مقابل الـ VS Code Extension
- ضبط CLAUDE.md والـ permissions والـ memory لسير عمل الفريق
- الاستعداد لاستخدام الذكاء الاصطناعي بإشراف في Phase 2

## Terminal (CLI) مقابل VS Code Extension

Claude Code يأتي بشكلين. استخدم الاثنين — يكملان بعضهما.

### CLI (Terminal Agent)

شغّله من أي مجلد مشروع:
```bash
npm install -g @anthropic-ai/claude-code
claude
```

**الأنسب لـ:**
- الـ refactors متعددة الملفات التي تمتد عبر ملفات كثيرة
- المهام المستقلة: "refactor هذا الـ controller، حدّث الـ tests، شغّل Pint"
- CI/CD scripting والأتمتة
- العمل عبر SSH أو في بيئات بدون واجهة رسومية
- المهام الطويلة حيث تصف الهدف وتدع الـ agent يعمل

### VS Code Extension

ثبّتها من الـ marketplace: **Claude Code** من Anthropic.

**الأنسب لـ:**
- التعديلات على ملف واحد مع inline diff preview
- طرح أسئلة حول الكود الذي تنظر إليه
- الإصلاحات السريعة حيث تريد مراجعة كل تغيير قبل القبول
- التفاعل بأسلوب pair-programming — احتكاك منخفض، تحكم عالٍ

### تشغيل الاثنين معاً

- الـ extension والـ CLI يمكنهما العمل في نفس المشروع في نفس الوقت
- يتشاركان نفس `CLAUDE.md` وسياق المشروع
- نصيحة: استخدم الـ extension للمهمة الحالية، والـ CLI لمهمة موازية في الخلفية

### مقارنة سريعة

| | **VS Code Extension** | **CLI (Terminal)** |
|--|----------------------|--------------------|
| Interface | IDE sidebar | Terminal |
| الأنسب لـ | Targeted edits، أسئلة، inline review | Long autonomous tasks، multi-file refactors |
| السياق | الملف الحالي + الملفات المفتوحة | المشروع بأكمله (تتحكم في النطاق) |
| التحكم | عالٍ — ترى كل تغيير | متغير — الـ agent يعمل باستقلالية |
| توافق المرحلة | Phase 2 (supervised) | Phase 3 (agentic) |

> ابدأ الفريق على **الـ extension** في Phase 2. قدّم **CLI agent** في Phase 3 مع تزايد الثقة.

## مرجع Slash Commands

الـ slash commands تُكتب مباشرة في prompt الـ Claude Code. تتحكم في الأوضاع، تدير الجلسات، وتشغّل workflows.

### الجلسة والتنقل

| Command | الوصف |
|---------|-------|
| `/help` | عرض جميع الأوامر المتاحة وطريقة الاستخدام |
| `/compact` | ضغط محفوظات المحادثة لتحرير مساحة الـ context window |
| `/clear` | مسح المحادثة بالكامل والبدء من جديد |
| `/status` | عرض معلومات الجلسة — الـ model، الـ mode، استهلاك الـ tokens |
| `/cost` | عرض استهلاك الـ tokens والتكلفة للجلسة الحالية |

### الـ Model والإعدادات

| Command | الوصف |
|---------|-------|
| `/model` | تغيير الـ model أثناء الجلسة (مثلاً `/model sonnet`، `/model opus`) |
| `/config` | فتح أو تعديل إعدادات Claude Code |

### الأوضاع

| Command | الوصف |
|---------|-------|
| `/plan` | دخول Planning Mode — Claude يستكشف ويحلل لكن لا يحرر الملفات |
| `/fast` | تفعيل/إلغاء الوضع السريع (نفس الـ model، مخرجات أسرع) |

### Coding Workflows

| Command | الوصف |
|---------|-------|
| `/commit` | عمل stage للتغييرات، توليد رسالة commit، وإنشاء git commit |
| `/review` | مراجعة تغييرات الكود أو pull request مع feedback تفصيلي |
| `/pr-comments` | عرض ومعالجة تعليقات مراجعة الـ PR |

### الملفات والسياق

| Command | الوصف |
|---------|-------|
| `/add-dir` | إضافة مجلد عمل إضافي لسياق الجلسة |
| `/release-notes` | توليد release notes من التغييرات الأخيرة |

### الأدوات والتكاملات

| Command | الوصف |
|---------|-------|
| `/mcp` | عرض MCP servers المتصلة وحالتها |
| `/listen` | الاستماع لمدخلات من مصادر خارجية |

### النظام

| Command | الوصف |
|---------|-------|
| `/login` | تسجيل الدخول لحساب Anthropic |
| `/logout` | تسجيل الخروج من الجلسة الحالية |
| `/doctor` | تشغيل فحوصات تشخيصية للتحقق من صحة Claude Code وإعداداته |
| `/bug` | الإبلاغ عن خطأ لفريق Claude Code |

> يمكنك أيضاً كتابة `/` في الـ prompt وسيعرض Claude Code اقتراحات autocomplete للأوامر المتاحة.

## Planning Mode

### ما هو؟

وضع read-only حيث Claude يستكشف ويحلل ويصمم خطة تنفيذ — لكنه **لا** يحرر أي ملفات. فكر فيه كـ "دع الذكاء الاصطناعي يدرس المشكلة قبل لمس أي شيء."

### متى تستخدمه؟

- قبل بدء refactor معقد — دع Claude يرسم خريطة الـ codebase أولاً
- القرارات المعمارية — اطلب من Claude الاستكشاف والتوصية
- عندما تريد تحليل بدون خطر التغييرات العرضية
- فهم كود غير مألوف — Claude يقرأ ويشرح بدون تعديل

### كيف يعمل؟

1. ادخل planning mode بـ `/plan` أو ابدأ بـ `claude --plan`
2. Claude يستطيع قراءة الملفات، البحث بـ grep/glob، تنفيذ أوامر read-only
3. Claude يخرج خطة منظمة بقوائم ملفات وخطوات تنفيذ
4. راجع الخطة، اطرح أسئلة، عدّل
5. عندما تكون راضياً، اخرج من planning mode لبدء التنفيذ

### مثال عملي

```
/plan How should we refactor the OrderController to use Action classes?
Show me which files are involved and what the migration steps would be.
```

Claude سيستكشف الـ codebase، يحدد كل الملفات المرتبطة، وينتج خطة خطوة بخطوة — بدون تغيير سطر واحد.

## Edit Mode — كيف يغيّر Claude الملفات

### دورة الموافقة/الرفض

كل تعديل ملف يقترحه Claude يمر بدورة موافقة:

1. Claude يقترح تغييراً كـ diff (ترى بالضبط ما سيتغير)
2. تستطيع **الموافقة** (تطبيق التغيير)، **الرفض** (تخطيه)، أو **طلب تعديل**
3. التعديلات المرفوضة لا تُطبَّق — Claude يستطيع المحاولة بأسلوب مختلف بناءً على ملاحظاتك

في **الـ VS Code Extension**، التغييرات تظهر كـ inline diffs في المحرر — تماماً مثل code review.

في **الـ CLI**، ترى terminal diff وتؤكد بـ `y` أو `n`.

### أوضاع الصلاحيات

| الوضع | السلوك | الأنسب لـ |
|-------|--------|----------|
| **Default** | يطلب إذن لكل تعديل وأمر | مرحلة التعلم، الكود الحساس |
| **Auto-edit** | يوافق تلقائياً على تعديلات الملفات، يسأل عن الأوامر | الـ refactors الموثوقة |
| **YOLO / Full auto** | يوافق تلقائياً على كل شيء | Phase 3، مشاريع مع CI safety nets |

### اختصارات لوحة المفاتيح (CLI)

| المفتاح | الإجراء |
|---------|---------|
| `y` / `Enter` | الموافقة على التغيير المقترح |
| `n` | رفض التغيير المقترح |
| `e` | تعديل التغيير المقترح قبل التطبيق |
| `Esc` | مقاطعة Claude أثناء الرد |
| `Shift+Tab` | التبديل بين إدخال سطر واحد ومتعدد الأسطر |

## Thinking Mode / Extended Thinking

### ما هو؟

Extended thinking يتيح لـ Claude التفكير في المشاكل المعقدة خطوة بخطوة قبل الرد. Claude يُظهر سلسلة التفكير الداخلية، مما ينتج إجابات أكثر عمقاً ودقة.

### متى تستخدمه؟

- القرارات المعمارية المعقدة مع tradeoffs متعددة
- تصحيح المشاكل الخفية حيث السبب غير واضح
- الـ refactors متعددة الخطوات حيث ترتيب العمليات مهم
- عندما تبدو إجابة Claude الأولى سطحية أو تفوّت edge cases

### كيف يعمل؟

- التفكير يُتحكم فيه عبر إعداد **effort** (انظر القسم التالي)
- effort أعلى = تفكير أكثر = نتائج أفضل للمشاكل الصعبة
- effort أقل = ردود أسرع للمهام البسيطة
- Claude يقرر مقدار التفكير بناءً على تعقيد المهمة عند ضبطه على `auto`

---

## Effort Levels

### ماذا تتحكم فيه؟

الـ effort يحدد مقدار التفكير الذي يطبقه Claude على كل رد. effort أعلى يعني extended thinking أكثر، نتائج أفضل للمهام المعقدة، لكن tokens أكثر وردود أبطأ.

### اختيار المستوى المناسب

| المستوى | الأنسب لـ | السرعة | تكلفة الـ Tokens |
|---------|----------|-------|-----------------|
| `auto` | دع Claude يقرر (الافتراضي، الموصى) | متغيرة | متغيرة |
| `low` | الأسئلة البسيطة، البحث السريع، الشروحات | سريع | منخفضة |
| `medium` | مهام البرمجة العادية، تعديلات ملف واحد | متوسط | متوسطة |
| `high` | الـ refactors المعقدة، المعمارية، debugging الصعب | بطيء | عالية |

### ضبط الـ Effort

في الـ CLI، اضغط `Shift+Tab` للوصول للإعدادات، أو:

```bash
claude config set effort high
```

> **قاعدة بسيطة:** اتركه على `auto`. Claude سيفكر أكثر عندما يحتاج. غيّر إلى `high` فقط عندما تعرف أن المهمة معقدة و Claude لا يعطي إجابات عميقة كفاية.

## CLAUDE.md — تعليمات المشروع

### ما هو؟

ملف markdown في root المشروع يقرأه Claude **تلقائياً** عند بداية كل جلسة. يحتوي على اتفاقيات الفريق، قواعد المعمارية، ومعايير البرمجة. فكر فيه كـ "مستندات onboarding لزميلك الذكاء الاصطناعي."

### ماذا تضع فيه؟

```markdown
# CLAUDE.md

## Project Overview
منصة تجارة إلكترونية مبنية بـ Laravel 11، PHP 8.2، MySQL 8.

## Architecture
- Controllers: thin، تفوّض إلى Action classes
- Business logic: app/Actions/{Domain}/{VerbNounAction}.php
- Validation: Form Requests، لا تعمل validate في controllers
- Models: relationships و scopes فقط، لا business logic

## Coding Standards
- اتبع PSR-12 عبر Laravel Pint
- استخدم ميزات PHP 8.2+ (enums، readonly، named arguments)
- Strict types في كل ملف: declare(strict_types=1)

## Testing
- استخدم Pest، ليس PHPUnit
- كل Action يجب أن يكون له unit test
- شغّل: ./vendor/bin/pest

## Important Rules
- لا تعدّل migrations تم تشغيلها في production
- استخدم دائماً database transactions للعمليات متعددة الخطوات
- لا تستخدم env() خارج ملفات config
```

### الموقع والتسلسل الهرمي

| الموقع | النطاق |
|--------|--------|
| Root `CLAUDE.md` | ينطبق على المشروع بأكمله |
| `app/Actions/CLAUDE.md` | ينطبق فقط داخل هذا المجلد (إضافي) |
| `tests/CLAUDE.md` | تعليمات خاصة بالاختبارات |

Claude يقرأ **جميع** ملفات CLAUDE.md المنطبقة على السياق الحالي — الـ root + أي ملفات على مستوى المجلدات الفرعية.

### اتفاقية الفريق

- **أودِع** CLAUDE.md في الـ repository — إنه إعداد مشترك للفريق
- **راجع** التغييرات على CLAUDE.md في الـ PRs — إنه يشكّل سلوك الذكاء الاصطناعي للجميع
- **حدّثه** مع تطور الاتفاقيات — إنه مستند حي

## Memory

### ما هو؟

معلومات مستمرة يتذكرها Claude **عبر الجلسات**، مخزنة محلياً على جهاز المطور. على عكس CLAUDE.md (المشترك عبر git)، الـ memory شخصي لكل مطور.

### كيف يعمل؟

- أخبر Claude: *"تذكر أن staging server لدينا على staging.example.com"*
- Claude يحفظ هذا في ملفات memory محلية
- في الجلسات المستقبلية، Claude يستدعيه تلقائياً
- تستطيع أن تطلب من Claude نسيان أشياء محددة

### Memory مقابل CLAUDE.md

| | **CLAUDE.md** | **Memory** |
|--|--------------|-----------|
| مشترك مع الفريق | نعم (مُودَع في git) | لا (محلي على جهازك) |
| الأنسب لـ | Coding standards، قواعد المعمارية | التفضيلات الشخصية، تفاصيل البيئة المحلية |
| يستمر عبر | كل المطورين، كل الجلسات | جلساتك فقط |
| يُحدَّث بواسطة | مراجعة PR | المحادثة |

> استخدم **CLAUDE.md** لكل ما يجب أن يتبعه الفريق. استخدم **memory** لسياقك الشخصي.

## Permissions

### نموذج الصلاحيات

Claude يطلب إذناً قبل تنفيذ الإجراءات المحتمل تدميرها. هذا قابل للضبط لكل tool ولكل مشروع.

### خيارات الإعداد

| الإعداد | التأثير |
|---------|---------|
| Per-tool approval | الموافقة على كل استخدام tool بشكل فردي (الأكثر أماناً) |
| Auto-allow reads | السماح بجميع عمليات القراءة بدون prompt |
| Auto-allow writes | السماح بتعديلات الملفات بدون prompt |
| Allow specific commands | مثلاً السماح بـ `npm test`، `./vendor/bin/pint` |
| Deny list | حظر tools أو أوامر محددة بالكامل |

### الإعداد الموصى حسب المرحلة

| المرحلة | قراءة الملفات | تعديل الملفات | تنفيذ الأوامر |
|---------|-------------|-------------|-------------|
| **Phase 1-2** | Auto-allow | يسأل كل مرة | يسأل كل مرة |
| **Phase 3 (مبكر)** | Auto-allow | Auto-allow | يسأل عن الأوامر الجديدة |
| **Phase 3 (ناضج)** | Auto-allow | Auto-allow | Auto-allow للأوامر الموثوقة |

### ضبط الصلاحيات

```bash
# السماح بعمليات القراءة
claude config set permissions.allow-read true

# السماح بأوامر محددة بدون prompt
claude config set permissions.allow-commands "npm test,./vendor/bin/pint,./vendor/bin/pest"
```

## Model Selection

### تغيير الـ Model أثناء الجلسة

```
/model                    # اختيار تفاعلي
/model opus               # التغيير إلى Opus
/model sonnet             # التغيير إلى Sonnet
/model haiku              # التغيير إلى Haiku (سريع، رخيص)
```

### متى تغيّر؟

| الموقف | الـ Model |
|--------|----------|
| الـ refactors المعقدة، القرارات المعمارية | **Opus** |
| الأسئلة السريعة، الشروحات، التعديلات البسيطة | **Sonnet** |
| التكرارات السريعة، المهام الحساسة للتكلفة | **Haiku** |

### مقارنة الـ Models

| | **Opus 4.6** | **Sonnet 4.6** | **Haiku 4.5** |
|--|-------------|---------------|--------------|
| Context window | 1M tokens | 200K tokens | 200K tokens |
| القوة | أعمق تفكير، الأفضل للمهام المعقدة متعددة الملفات | توازن ممتاز بين السرعة والجودة | الأسرع، الأرخص |
| Agentic capability | Full | Full | Full |
| الأنسب لـ | القرارات المعمارية، الـ refactors المعقدة، debugging متعدد الخطوات | البرمجة اليومية، الميزات العادية، code review | الأسئلة السريعة، التعديلات البسيطة، الشروحات |
| السرعة | أبطأ | سريع | الأسرع |
| التكلفة | الأعلى | متوسطة | الأقل |
| متى تستخدمه | المهمة بها 3+ خطوات، تمس 5+ ملفات، أو تحتاج تفكير عميق | الافتراضي للعمل اليومي | البحث السريع، التغييرات البسيطة |

### الإعداد الافتراضي

```bash
claude config set model claude-opus-4-6      # الافتراضي Opus
claude config set model claude-sonnet-4-6    # الافتراضي Sonnet
```

> مرجع متقاطع: انظر [1.3 اختيار الأدوات](03-tooling-selection.md) لجدول مقارنة الـ models الكامل.

## MCP (Model Context Protocol)

### ما هو؟

MCP هو protocol يتيح لـ Claude الاتصال بـ **أدوات ومصادر بيانات خارجية**. فكر فيه كـ "plugins" لـ Claude Code — يوسّع ما يستطيع Claude الوصول إليه خارج الملفات المحلية.

### حالات الاستخدام لفرق Laravel

| MCP Server | ماذا يوفر |
|------------|----------|
| Database | Schema-aware queries، فحص بنية الجداول |
| Notion / Linear / Jira | قراءة tickets، تحديث الحالة، ربط PRs |
| Sentry | قراءة تقارير الأخطاء، فهم مشاكل production |
| Playwright | Browser automation للاختبارات |
| Custom internal APIs | توصيل Claude بخدماتك الداخلية |

### الإعداد

MCP servers تُعرَّف في ملف إعدادات Claude Code:

```json
{
  "mcpServers": {
    "database": {
      "command": "npx",
      "args": ["@anthropic-ai/mcp-server-postgres", "postgresql://..."]
    }
  }
}
```

### اعتبارات الأمان

- اتصل فقط بـ MCP servers تثق بها
- راجع ما هي الصلاحيات التي يمنحها كل server
- الـ MCP servers تعمل محلياً — لا ترسل بياناتك إلى Anthropic
- استخدم read-only database connections للأمان

## Hooks

### ما هي؟

Scripts تعمل تلقائياً **قبل أو بعد** استخدام Claude لأداة. مفهوم مشابه لـ git hooks، لكن لتنفيذ أدوات Claude.

### حالات الاستخدام

| Hook Trigger | مثال الاستخدام |
|-------------|---------------|
| Before file edit | التحقق أن الملف المستهدف ليس محمياً |
| After file edit | Auto-format بـ Pint بعد كل تعديل ملف PHP |
| Before shell command | التحقق أن الأمر آمن قبل التنفيذ |
| After shell command | تسجيل جميع الأوامر التي نفذها Claude للمراجعة |

### الإعداد

Hooks تُعرَّف في إعدادات Claude Code:

```json
{
  "hooks": {
    "postToolUse": [
      {
        "tool": "edit",
        "command": "./vendor/bin/pint $FILE"
      }
    ]
  }
}
```

### لماذا Hooks مهمة؟

- تطبيق المعايير تلقائياً — حتى عندما ينسى Claude
- مسار المراجعة — تسجيل ما يفعله Claude لمراجعة الفريق
- شبكة أمان — حظر العمليات الخطيرة قبل حدوثها

## Git Worktrees

### ما هي؟

Git worktrees تتيح لك checkout **عدة branches في وقت واحد** في مجلدات منفصلة. Claude Code يستطيع العمل في worktree لعمل branch معزول بدون التأثير على working directory الرئيسي.

### لماذا تستخدمها مع Claude Code؟

- اعمل على feature بنفسك بينما Claude يعمل refactor في worktree آخر
- لا branch switching — لا stashing — لا فقدان سياق
- كل worktree له working directory خاص لكنه يتشارك git history

### الإعداد

```bash
# إنشاء worktree لـ feature branch
git worktree add ../my-project-feature feature/PROJ-101-new-feature

# تشغيل Claude في الـ worktree المعزول
cd ../my-project-feature
claude
```

### متى تستخدمها؟

- مهام AI متوازية على features مختلفة
- مراجعة عمل Claude على branch بدون مغادرة عملك الحالي
- Phase 3: تشغيل عدة Claude agents على features مختلفة بالتزامن

## Subagents

### ما هي؟

Claude يستطيع إطلاق **مهام فرعية** تعمل بالتوازي. كل subagent يعالج جزءاً محدداً من مهمة أكبر بشكل مستقل.

### كيف تعمل؟

1. Claude يحدد أجزاء المهمة التي يمكن تشغيلها بالتوازي
2. كل subagent يعمل بشكل مستقل (مثلاً "استكشف auth module" بينما آخر "يستكشف payment module")
3. النتائج تُجمَع وتُدمَج
4. لا تزال توافق على التغييرات النهائية

### متى تُفعَّل؟

- مهام البحث والتحليل واسعة النطاق عبر codebase كبير
- استكشاف عدة مناطق مستقلة بالتزامن
- المهام حيث التوازي آمن (لا تعارضات ملفات بين المهام الفرعية)

> لا تحتاج لضبط subagents — Claude يستخدمها تلقائياً عندما تكون مفيدة.

## Context Management

### كيف يعمل الـ Context Window

- Claude لديه context window محدود (1M tokens لـ Opus، 200K لـ Sonnet)
- كل رسالة، قراءة ملف، ومخرجات tool تستهلك tokens
- عندما يمتلئ الـ context، التفاصيل السابقة تُضغط تلقائياً

### /compact — استرجاع الـ Context

أمر `/compact` يضغط محفوظات المحادثة:

- يحتفظ بالمعلومات الأساسية (القرارات، المهمة الحالية، النتائج الرئيسية)
- يتخلص من مخرجات الأدوات المطوّلة والخطوات الوسيطة
- استخدمه عندما يبدأ Claude بتكرار نفسه أو يفقد التتبع

### أفضل الممارسات

| الممارسة | لماذا |
|----------|-------|
| ابدأ مركّزاً — أخبر Claude بالضبط ما تحتاجه | يتجنب إهدار الـ context على الاستكشاف |
| تجنب قراءة ملفات غير ضرورية | كل قراءة ملف تكلف tokens |
| استخدم `/compact` بشكل استباقي خلال الجلسات الطويلة | يمنع فيضان الـ context |
| قسّم المهام الكبيرة جداً إلى عدة جلسات | أفضل من جلسة واحدة محمّلة زيادة |
| اجعل CLAUDE.md مختصراً | يُعاد قراءته عند بداية كل جلسة |
| استخدم `/clear` للبدء من جديد تماماً | عندما تكون المحادثة انحرفت كثيراً |

## Multi-file Operations

### كيف يعالج Claude الـ Refactors المعقدة

1. Claude يقرأ الملفات المعنية ويبني نموذجاً ذهنياً للكود
2. يخطط التغييرات عبر جميع الملفات (خاصة في Planning Mode)
3. يقترح تعديلات ملف بملف — تراجع كل diff
4. يتتبع التبعيات بين الملفات (مثلاً إعادة تسمية method تحدّث جميع مواقع الاستدعاء)

### أفضل الممارسات

- **أعطِ نطاقاً واضحاً:** "Refactor الـ User module" وليس "أصلح كل شيء"
- **استخدم CLAUDE.md** لتعريف المعمارية المستهدفة — Claude يتبعه
- **راجع multi-file diffs بعناية** — تحقق من الحواف (imports، type hints، references)
- **للـ refactors الكبيرة جداً:** استخدم Planning Mode أولاً، ثم نفّذ على مراحل
- **شغّل الـ tests بعد كل مرحلة** — اكتشف المشاكل مبكراً
