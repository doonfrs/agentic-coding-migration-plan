# 3.1 منهجية الوكيل

> في Phase 2 استخدم المطور الذكاء الاصطناعي ليكتب كوداً أسرع. في Phase 3 يتوقف المطور عن كتابة معظم الكود ويبدأ بتوجيه الـ agents - يراجع النتائج، لا ضربات المفاتيح. هذا ينجح فقط لأن Phase 1 و Phase 2 بنتا المعايير وشبكات الأمان التي تتيح لك الثقة بمخرَج الـ agent دون مراقبة كل سطر يكتبه.

---

## الأهداف

- تعريف حلقة تطوير وكيلية قابلة للتكرار يستطيع الفريق تشغيلها على أي ticket
- إسناد أدوار agents واضحة بحيث يكون لكل خطوة في الحلقة مالك
- تعريف الـ human checkpoints حيث يجب على شخص أن يوافق قبل أن يتقدم العمل
- جعل المنهجية قابلة للقياس، حتى يمكن تتبّع التحسّن (انظر [3.3 Team Enablement](03-team-enablement.md))

---

## ما الذي يتغيّر في Phase 3

التحوّل في *أين يصرف الإنسان انتباهه*. الميكانيكا المبنية في المراحل السابقة لا تتغيّر - تُوجَّه نحو agent مستقل بدلاً من agent مُشرَف عليه.

| | Phase 2 (بإشراف) | Phase 3 (وكيلي) |
|--|------------------|------------------|
| من يكتب الكود | المطور، بمساعدة AI | الـ agent |
| من يشغّل الاختبارات | المطور | الـ agent (ثم CI) |
| الأداة | VS Code extension | Claude Code CLI agent |
| تركيز الإنسان | كل سطر، أثناء كتابته | الخطة والنتيجة النهائية |
| دور الإنسان | مؤلّف | موجّه ومراجِع |

كل ما يجعل هذا آمناً موجود أصلاً: branch protection من [1.4 Git Workflow](../phase1/04-git-workflow.md)، الـ test suite من [2.1 Testing](../phase2/01-testing.md)، الـ [CI Pipeline](../phase2/02-ci-pipeline.md) الذي يحرس الدمج، وغرائز المراجعة من [2.4 Supervised AI](../phase2/04-supervised-ai.md). Phase 3 تربطها في حلقة.

## تصميم الـ Agentic SDLC

دورة حياة تطوير البرمجيات الوكيلية هي حلقة ببوابتين بشريتين - واحدة في البداية (الموافقة على الخطة) وواحدة قبل الدمج (مراجعة النتيجة). كل ما بينهما، يفعله الـ agent.

```
   ┌──────────────────────────────────────────────────────────┐
   │                                                          │
   │   1. Ticket / task                                       │
   │          │                                               │
   │          ▼                                               │
   │   2. Agent proposes a PLAN  ──▶  👤 Human approves plan   │  ◀── checkpoint 1
   │          │                                               │
   │          ▼                                               │
   │   3. Agent implements                                    │
   │          │                                               │
   │          ▼                                               │
   │   4. Agent runs tests locally  ──▶ fail? back to step 3  │
   │          │ pass                                          │
   │          ▼                                               │
   │   5. Agent pushes branch, opens PR                       │
   │          │                                               │
   │          ▼                                               │
   │   6. CI pipeline runs  ──▶ fail? agent reads errors,     │
   │          │ pass            fixes, pushes again           │
   │          ▼                                               │
   │   7. 👤 Human reviews & approves  ◀── checkpoint 2       │
   │          │                                               │
   │          ▼                                               │
   │   8. Merge to main  ──▶  deploy                          │
   │                                                          │
   └──────────────────────────────────────────────────────────┘
```

كل خطوة تُقابِل أصلاً بُني في مرحلة سابقة:

| الخطوة | مدعومة بـ |
|--------|-----------|
| اقتراح الخطة | Plan mode + `CLAUDE.md` ([1.5 Claude Code](../phase1/05-claude-code.md)) |
| التنفيذ | الـ agent + المعايير من [1.2 Laravel Standards](../phase1/02-laravel-standards.md) |
| الاختبارات المحلية | Pest suite ([2.1 Testing](../phase2/01-testing.md)) |
| الـ Push و PR | Git workflow + branch protection ([1.4](../phase1/04-git-workflow.md)) |
| بوابة CI | [2.2 CI Pipeline](../phase2/02-ci-pipeline.md) |
| المراجعة البشرية | قواعد المراجعة من [2.4 Supervised AI](../phase2/04-supervised-ai.md) |

> **الـ agent لا يدمج كوده أبداً.** الخطوة 8 تحدث فقط بعد أن يوافق إنسان في الخطوة 7 ويكون CI أخضر. الحلقة مستقلة في الوسط ومحروسة عند الطرفين.

## أدوار الوكلاء (Agent Roles)

"وكيلي" لا يعني أن agent واحداً يفعل كل شيء في مرور واحد. العمل ينقسم إلى أدوار - تُنفَّذ كـ Claude Code modes أو subagents أو skills (انظر [1.5 Claude Code Deep Dive](../phase1/05-claude-code.md)). الـ ticket الواحد يمرّ عبرها جميعاً.

| الدور | المسؤولية | المُدخَل | المُخرَج |
|-------|-----------|----------|----------|
| **Planner** | تفكيك الـ ticket إلى approach ملموس | Ticket + context الكود | خطة قابلة للمراجعة |
| **Implementer** | كتابة الكود ليطابق الخطة المعتمَدة | الخطة المعتمَدة | تغييرات الكود |
| **Tester** | كتابة وتشغيل الاختبارات؛ التحقق من السلوك | تغييرات الكود | test suite ناجح |
| **Reviewer (الناقد)** | مراجعة ذاتية للأخطاء والمعايير والأمان قبل الـ PR | الـ diff | مشاكل للإصلاح، أو "جاهز" |

دور **Reviewer** هو ما يجعل الحلقة جديرة بالثقة. قبل أن يرى إنسان الـ PR أصلاً، يكون agent ناقد قد فحص الـ diff مقابل المعايير وشغّل `/security-review` على المناطق المقيّدة - حتى يراجع الإنسان عملاً مصقولاً، لا مسوّدات أولى.

> **الأدوار فصل للاهتمامات، لا بالضرورة جلسات منفصلة.** agent واحد قادر يستطيع لعب كل دور بالتسلسل؛ القيمة هي أن كل اهتمام ينال انتباهاً صريحاً، بنفس الطريقة التي يخطط بها مطور جيد، ثم يبرمج، ثم يختبر، ثم يراجع.

## نقاط التحقق البشرية (Human Checkpoints)

الاستقلال بلا checkpoints هو مجرد ذكاء اصطناعي بلا إشراف بخطوات إضافية. هذه هي النقاط التي يجب أن ينخرط فيها إنسان - مختارة لأنها حيث يعيش الحكم، أو المسؤولية، أو عدم قابلية التراجع.

| نقطة التحقق | المالك | ماذا يتحقق | البوابة |
|--------------|--------|-------------|----------|
| الموافقة على الخطة | المطور / lead | أن الـ approach سليم قبل كتابة أي كود | الـ agent ينتظر الموافقة |
| مراجعة الـ PR | المراجِع | الكود صحيح، مقروء، يتّبع المعايير | موافقة مطلوبة على الـ PR |
| توقيع المنطقة المقيّدة | senior | auth، payments، migrations، الأمان صحيحة | موافقة senior إلزامية |
| نشر الإنتاج | lead | الـ release جاهز والـ rollback جاهز | موافقة نشر يدوية |

### المبدأ

**البشر يراجعون النية والنتائج؛ الأتمتة تفرض الصحة.**

- *الصحة* - التنسيق، الأنواع، الاختبارات، الـ coverage - ميكانيكية. CI يفعلها، كل مرة، دون كلل. لا تصرف انتباه إنسان هناك.
- *النية والحكم* - هل هذا الـ approach الصحيح؟ هل هذا آمن للشحن؟ هل هذا التغيير المقيّد يفعل فعلاً ما نقصده؟ - بشري. هناك توجد الـ checkpoints.

هذا هو نفس الانضباط من [2.4 Supervised AI](../phase2/04-supervised-ai.md)، مُرسمَناً في نظام: أنت تملك الكود الذي تدمجه، مهما كتبه. الحدود والسياسات والـ KPIs التي تبقي هذا صادقاً على نطاق واسع مغطّاة في [3.4 Governance](04-governance.md).

## قائمة التحقق - اكتمل عند

- [ ] الحلقة الوكيلية موثّقة والفريق شغّلها من البداية للنهاية على ticket حقيقي
- [ ] الـ agent يقترح خطة وينتظر موافقة الإنسان قبل التنفيذ
- [ ] الـ agent يشغّل الاختبارات محلياً ويكرّر على إخفاقات CI بنفسه
- [ ] لا يُدمج أي PR من تأليف الـ agent بدون CI أخضر **و** موافقة إنسان
- [ ] أدوار الـ agents (Planner / Implementer / Tester / Reviewer) مُعرَّفة ومستخدَمة
- [ ] الـ human checkpoints الأربع كلها مفروضة، مع المناطق المقيّدة تتطلب توقيع senior
- [ ] نشر الإنتاج لا يزال يتطلب موافقة إنسان صريحة
- [ ] الفريق يفهم أنه يملك الكود المدموج بغض النظر عمن ألّفه
