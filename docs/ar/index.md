# خطة الانتقال إلى البرمجة الوكيلية

> خطة منظمة لمدة 3 أشهر للانتقال التدريجي بفريق Laravel من أسلوب التطوير التقليدي غير المقنن إلى البرمجة بمساعدة الذكاء الاصطناعي — دون المساس بما هو موجود في الإنتاج.

> الذكاء الاصطناعي يضخّم ما هو موجود. إذا كان الكود بلا معايير، سينتج الوكيل المزيد من نفس الفوضى. المعايير تأتي أولاً.

- [1.2 معايير Laravel](./phase1/02-laravel-standards.md)

## أسلوب العمل

- اجتماعان أسبوعيان للإشراف (مراجعة التقدم، رفع العقبات، تدريب عملي)
- الجودة والاستقرار قبل السرعة — لا اختصارات
- مزيج من الجلسات النظرية وتطبيق المهام الحقيقية
- تدخل مباشر عند ظهور عوائق
- كل مرحلة تنتهي بجلسة مراجعة للفريق قبل الانتقال للتالية

## لماذا هذا الترتيب؟

| الخطر | الحل |
|-------|------|
| الذكاء الاصطناعي يعزز أنماط الكود السيئة | المعايير تُطبَّق في الشهر الأول قبل تقديم الذكاء الاصطناعي |
| أخطاء الذكاء الاصطناعي تصل إلى الإنتاج | الاختبارات + بوابة CI تُقدَّم في الشهر الثاني قبل توسيع استخدام الذكاء الاصطناعي |
| الفريق يتبنى الذكاء الاصطناعي بدون إشراف | مرحلة إشراف قبل سير عمل الوكيل المستقل |
| لا توجد طريقة لقياس التحسن | مؤشرات الأداء وخطوط الأساس تُسجَّل قبل بدء المرحلة الوكيلية |
