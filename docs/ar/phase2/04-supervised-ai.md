# 2.4 تقديم الذكاء الاصطناعي بإشراف

> تعامل مع الذكاء الاصطناعي كمطور junior لامع وسريع ومتحمّس قرأ كل توثيق Laravel لكنه لم يُكوَ أبداً بحادثة إنتاج. سينتج كثيراً من الكود الجيد وكارثة خاطئة بثقة من حين لآخر. الإشراف هو ما يحوّل ذلك الـ junior إلى شخص يثق به الفريق.

---

## الأهداف

- جعل كل مطور يستخدم Claude Code على مهام حقيقية، تحت إشراف
- وضع حدود واضحة: أي المهام يستطيع الذكاء الاصطناعي تولّيها، وأيها لا
- بناء أساس من مهارة prompt engineering عبر الفريق
- ترسيخ عادة مراجعة غير قابلة للتفاوض للكود المولَّد بالذكاء الاصطناعي
- بناء الثقة والانضباط اللذين يعتمد عليهما استقلال Phase 3

---

## لماذا "بإشراف"

شبكات الأمان الآن في مكانها: [الاختبارات](01-testing.md)، و [CI pipeline](02-ci-pipeline.md)، و [بيئة](03-environment.md) متسقة. الفريق يعرف الأداة أيضاً - [1.5 Claude Code Deep Dive](../phase1/05-claude-code.md) غطّى ميزاتها. هذه المرحلة هي حيث تلتقي النظرية بكود حقيقي.

تُسمّى *بإشراف* عن قصد. في Phase 2، يبقى الإنسان دائماً في الحلقة: يختار المهمة، يوجّه الـ prompt، ويراجع كل سطر قبل دمجه. الذكاء الاصطناعي متعاون، لا agent مستقل. ذلك يأتي في [Phase 3](../phase3/01-agentic-methodology.md) - وفقط بعد أن يبني الفريق الغرائز التي تعلّمها هذه المرحلة.

> **ابدأ على الـ extension، لا على CLI agent.** كما هو موصى به في [1.3 Tooling Selection](../phase1/03-tooling-selection.md)، الـ VS Code extension يبقي المطور في مقعد القيادة. الـ CLI agent المستقل أداة Phase 3.

## مهام آمنة للذكاء الاصطناعي (AI-Safe Tasks)

هذه مهام يكون فيها الذكاء الاصطناعي قوياً فعلاً ويكون نطاق الضرر من الخطأ صغيراً. تحصل على **مراجعة خفيفة** - اقرأها، شغّل الاختبارات، ادمجها.

| نوع المهمة | لماذا هي آمنة | مستوى المراجعة |
|------------|----------------|------------------|
| Boilerplate CRUD | قابل للتنبؤ، قائم على pattern | خفيف |
| كتابة اختبارات لكود موجود | الاختبارات قابلة للتحقق؛ تقرأ الـ assertions | خفيف + اقرأ الـ assertions |
| Refactors **مع test coverage** | الاختبارات تلتقط الـ regressions | خفيف |
| التوثيق والتعليقات | لا أثر على الـ runtime | خفيف |
| Blade views / markup الواجهة | قابل للتحقق بصرياً | خفيف |
| Form Requests و validation rules | سهل القراءة والاختبار | خفيف |
| Factories و seeders | كود للاختبار فقط | خفيف |
| تعديلات متكررة عبر ملفات كثيرة | الذكاء الاصطناعي يبرع في الاتساق الميكانيكي | خفيف |

النمط: **الآمن للذكاء الاصطناعي يعني القابل للتحقق.** إذا استطاع إنسان أن يؤكد بسرعة أن النتيجة صحيحة (أو أكّدها اختبار)، يمكن للذكاء الاصطناعي أن يقود.

## مناطق مقيّدة على الذكاء الاصطناعي (AI-Restricted Areas)

هذه هي المناطق التي رُسمت كعالية الخطورة في [1.1 Codebase & Architecture Review](../phase1/01-codebase-architecture.md). يمكن للذكاء الاصطناعي أن *يساعد* هنا، لكن يجب على senior أن يراجع ويوقّع. لا مراجعة خفيفة أبداً.

| المنطقة المقيّدة | لماذا | المطلوب |
|-------------------|-------|----------|
| Authentication & authorization | خطأ دقيق = ثغرة أمنية | توقيع senior |
| Payments & billing | المال لا يسامح | توقيع senior |
| Database migrations على بيانات الإنتاج | صعب أو مستحيل العكس | مراجعة senior + backup |
| كود حسّاس أمنياً (encryption، tokens، file uploads) | أنماط فشل عالية الأثر | senior + [`/security-review`](#review-rules) |
| Infrastructure و queue configuration | يؤثر على النظام كله | توقيع senior |
| أي كود بلا test coverage | لا شيء يلتقط الـ regression | اكتب الاختبارات أولاً، ثم تابع |

> **قاعدة عملية:** إذا كان الخطأ هنا سيكلّف مالاً، أو يسرّب بيانات، أو يصعب التراجع عنه، فإن إنساناً يملكه. الذكاء الاصطناعي يستطيع المسوّدة؛ والـ senior يقرر.

## الجلسات العملية (Hands-on Sessions)

هذه المرحلة عملية، لا محاضرة. خطة الترحيل تتضمن أصلاً اجتماعَي إشراف أسبوعيين - استخدمهما لتدريب عمل الذكاء الاصطناعي على tickets حقيقية.

### هيكل الجلسة

| الجزء | المدة | التركيز |
|-------|-------|----------|
| الإحماء | 10 دقائق | مراجعة PRs الأسبوع الماضي المساعَدة بالذكاء الاصطناعي معاً |
| Live coding | 30 دقيقة | الـ lead يختار ticket حقيقياً، والفريق يقود Claude Code عليه |
| Pairing | 30 دقيقة | المطورون يعملون أزواجاً على tickets خاصة بهم مع AI؛ الـ lead يتنقّل |
| الخلاصة | 10 دقائق | ما نجح، ما أخطأ فيه AI، ما يجب مراقبته |

### تدرّج المهام

ابدأ صغيراً وقابلاً للتحقق، ثم وسّع النطاق مع بناء الثقة:

1. **الأسبوع 1-2:** مهام AI-safe فقط - اكتب اختباراً، أضِف form request، ولّد factory
2. **الأسبوع 3:** ميزة CRUD صغيرة من البداية للنهاية، مراجَعة بالكامل
3. **الأسبوع 4:** refactor على كود لديه اختبارات أصلاً

ارجع إلى [1.5 Claude Code Deep Dive](../phase1/05-claude-code.md) للميزات المحددة (plan mode، slash commands، `CLAUDE.md`) التي يجب على الفريق التمرّن عليها في هذه الجلسات.

## أساسيات Prompt Engineering

جودة مخرجات الذكاء الاصطناعي تتحدد في معظمها بجودة الـ prompt. الفريق لا يحتاج حيلاً - يحتاج بضع عادات صلبة.

### العادات التي تهم

- **أعطِ context.** وجّه الذكاء الاصطناعي إلى الملفات ذات الصلة، الـ model المرتبط، الـ pattern الموجود الذي يجب اتّباعه.
- **كن محدداً.** "أضف validation" ضعيف. "أضف Form Request يتحقق أن `email` مطلوب وفريد على `users`" قوي.
- **اطلب خطة أولاً.** في أي شيء غير تافه، اجعل الذكاء الاصطناعي يقترح approach قبل أن يكتب كوداً. إصلاح خطة أرخص من إصلاح diff.
- **حدّد acceptance criteria.** أخبره كيف يبدو "منجَز" - مثالياً كاختبار يجب أن يجعله ينجح.
- **كرّر.** أول مخرَج هو مسوّدة. حسّنه؛ لا تقبل-وادمج.

### prompt ضعيف مقابل قوي

```text
❌ ضعيف:
"Make an orders feature."

✅ قوي:
"Add an OrderController with store() following the pattern in
ProductController. Validate via a new StoreOrderRequest (product_id
required+exists, quantity required+min:1). Return 201 with the order
JSON. Write a Pest feature test covering the happy path and a 422
validation case. Show me the plan before editing."
```

> **تذكّر `CLAUDE.md`.** الـ `CLAUDE.md` الخاص بالمشروع (من [1.5](../phase1/05-claude-code.md)) يغذّي الذكاء الاصطناعي أصلاً بمعاييرك واصطلاحاتك في كل prompt. الـ prompts الجيدة تبني على ذلك الـ context المشترك بدلاً من تكراره.

## قواعد المراجعة (Review Rules)

هذا قلب الإشراف. الكود المولَّد بالذكاء الاصطناعي *غير مُراجَع افتراضياً* - المراجِع البشري هو شبكة الأمان التي لا تستطيع الاختبارات و CI استبدالها.

### غير القابلة للتفاوض

- ❌ **لا تدمج أبداً كوداً لم تقرأه.** "الاختبارات نجحت" ضروري، لا كافٍ.
- ❌ **لا تدمج أبداً اختبارات كتبها AI بدون قراءة كل assertion.** الذكاء الاصطناعي رائع في *هيكل* الاختبار وضعيف في معرفة *السلوك الصحيح* للـ business (انظر [2.1 Testing](01-testing.md)).
- ✅ **شغّله محلياً.** لا تراجع الـ diff فقط - شغّل الكود.
- ✅ **CI يجب أن يكون أخضر.** Pint و PHPStan و Pest كلها تنجح (انظر [2.2 CI Pipeline](02-ci-pipeline.md)). لا استثناءات.
- ✅ **شغّل `/security-review` على أي شيء يمسّ منطقة مقيّدة** - auth، authorization، payments، أو user input.

### أسئلة المراجِع

| السؤال | لماذا يهم |
|--------|-----------|
| هل أفهم كل سطر؟ | إذا لم تستطع شرحه، لا تستطيع امتلاكه |
| هل يتّبع patterns الموجودة لدينا؟ | الذكاء الاصطناعي يخترع أحياناً اصطلاحاته الخاصة |
| هل assertions الاختبار صحيحة فعلاً؟ | اختبار ناجح قد يؤكّد الشيء الخاطئ |
| هل مسّ أي شيء في منطقة مقيّدة؟ | إن فعل، صعّد إلى مراجعة senior |
| هل كنت سأشحن هذا لو كتبته بنفسي؟ | المعيار الصادق لـ "جيد بما يكفي" |

> **قاعدة عملية:** أنت تملك الكود الذي تدمجه، بغض النظر عمن أو ما كتبه. "الذكاء الاصطناعي كتبه" ليس أبداً تفسيراً لخطأ إنتاج. الإشراف هنا هو بالضبط العضلة التي تحوّلها Phase 3 إلى [human checkpoints](../phase3/01-agentic-methodology.md) في agentic coding.

## قائمة التحقق - اكتمل عند

- [ ] كل مطور أنجز PR واحداً على الأقل بمساعدة الذكاء الاصطناعي تحت إشراف
- [ ] حدود AI-safe مقابل AI-restricted موثّقة ومتّفق عليها
- [ ] الفريق يشغّل Claude Code عبر الـ VS Code extension على tickets حقيقية
- [ ] كل مطور يستطيع كتابة prompt قوي وغني بالـ context وطلب خطة أولاً
- [ ] لا يُدمج أي كود مولَّد بالذكاء الاصطناعي بدون قراءة إنسان لكل سطر
- [ ] الاختبارات المكتوبة بالذكاء الاصطناعي تُراجَع assertion بـ assertion قبل الدمج
- [ ] `/security-review` يُشغَّل على كل PR يمسّ منطقة مقيّدة
- [ ] الجلستان العمليتان الأسبوعيتان تعملان وتنتجان PRs مراجَعة
